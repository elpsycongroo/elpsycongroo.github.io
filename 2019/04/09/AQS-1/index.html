AQS源码分析（ReentrantLock - lock() & unlock())

本文主要介绍AQS的基本结构和属性，以及它在java并发包SDK中可重入锁(ReentrantLock)中的实现，主要解析加锁和解锁的源码部分，帮助我们更好的理解AQS的作用，以及为什么AQS对于并发包如此重要。

AQS的结构

AQS是一个Node组成的双向链表。含有以下属性：

  1. Node head - 头结点，AQS的阻塞队列不包含头结点，可以理解为当前持有锁的线程。
  2. Node tail - 阻塞队列的尾结点
  3. int state - 代表当前锁的状态，state>0 代表有线程持有当前锁，state>1代表锁重入了
  4. Thread exclusiveOwnerThread - 用来代表当前持有锁的线程，比如判断当前线程是否等于它来判断是否重入。

总结一下几个分支吧。

（一）
线程争抢锁->抢到了->设置锁的owner线程为自身->执行（结局1）

线程争抢锁->没抢到（try acquire失败），准备进入阻塞队列->进入acquireQueued(addWaiter(Node.EXCLUSIVE), arg))方法->进入addWaiter(Node node)方法

此时有以下情况：

  1. 试着快速入队，如果成功，加入队伍末尾，返回
  2. 退化到自旋入队，入队成功，返回

（总之都入队成功了）

（二）
入队成功后，进入acquireQueued(addWaiter(Node.EXCLUSIVE), arg))方法后有以下情况：

  1. 前驱是头结点，直接尝试抢锁，抢到且线程没有中断->无需阻塞（结局2）
  2. 前驱不是头结点，或者没有抢到锁->检查是否需要挂起该线程

（三）
在情况（二）2.的情况下，执行if(shouldParkAfterFailedAcquire(p, node) &&parkAndCheckInterrupt()), 首先看第一个方法的结果：

  1. 当前节点的前驱节点为waitStatus=-1, return true（可以直接挂起）
  2. 当前结点的前驱节点waitStatus>0即已经放弃排队，向前寻找直到一个状态<=0的节点，将其和自身双向链接。返回false继续下一个循环。如果前驱节点状态为-1，下个循环跳到1，否则进行下一步。
  3. 当前节点的前驱节点状态为0或-3，尝试利用CAS将其设置为-1，返回false继续下一个循环。

这里的循环指的是acquireQueued中的for(;;)

（四）
shouldParkAfterFailedAcquire返回true，执行parkAndCheckInterrupt()方法，顾名思义，该方法会执行LockSupport.park()方法阻塞目标线程，同时检查目标线程是否被中断了。

  1. 线程没有被中断，shouldParkAfterFailedAcquire返回true, parkAndCheckInterrupt()返回false, 线程被挂起阻塞等待唤醒（结局3）
  2. 线程被中断，shouldParkAfterFailedAcquire返回true, parkAndCheckInterrupt()返回true, 则acquireQueued()方法会返回true,最后回到acquire()方法中执行selfInterrupt()，中断线程（结局4）。

总结

AQS在ReentrantLock中的实现主要是依靠states、阻塞队列和队列中线程的阻塞和唤醒。

锁的状态依赖于states.states=0的时候，代表可以被争抢。states>0的时候代表有线程抢到了重入锁。之后该线程再次进入锁的时候只需要states++就可以，释放锁的时候states–，直到states=0代表完全释放。

只有发生锁竞争的时候，才会用到阻塞队列，此时头结点才会在线程进入阻塞队列时被初始化为new Node()。此时waitStatus=0，在shouldParkAfterFailedAcquire()方法中才会被修改为-1.该方法在后续节点加入时都会修改前驱节点的waitStatus.

队列中的头节点实际不属于阻塞队列，它可以优先CAS来设置states=1抢到锁，此时通过设置exclusiveOwnerThread来标明锁的所有权，之后通过查询该属性来判断线程是否是这把锁的主人。

AQS中的阻塞队列是一个FIFO的双向链表，每个节点在加入的时候都会加入到链表的末尾，并将前驱设置为waitStatus=-1的节点。

AQS中阻塞队列的挂起（阻塞）是使用了LockSupport.park(thread)，唤醒则是用unpark.队列中每个节点依赖于前驱节点来唤醒，其waitStatus必须为-1.而在唤醒后，节点的waitStatus被修改为0.

调用了unlock后，头结点唤醒它的后继节点。特殊情况下，从后向前寻找最靠前的waitStatus<=0的节点，此时该节点的前驱为头结点，可以争抢锁了。
