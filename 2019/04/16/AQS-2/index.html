区别1：公平锁在lock()方法中是直接acquire(1)的

区别2：在非公平锁的tryAcquire方法中，如果锁的states=0，则直接CAS，成功了获取到锁；而公平锁中则还要判断是否有线程在队列中等待，没有才可以CAS。

如果这两次CAS都失败了，那么公平锁和非公平锁没有任何区别。

相对而言， 非公平锁性能更好，吞吐量更大，但是获取锁的时间不确定，容易出现线程饥饿问题。

Condition

在Java的管程中，只隐含了一个Condition队列；而并发包SDK中可以指定多个Condition，这也是并发包SDK灵活的地方。每个Condition都有一个阻塞队列，经常可以用在生产者-消费者模型中。

在使用Condition时，必须拥有相应的锁。和管程模型中类似，必须持有Monitor的锁。

我们在阻塞队列中常用的一种就是ArrayBlockingQueue，它可以实现生产者-消费者场景，Doug Lea给出了一个用Condition实现的例子：

使用WAIT-NOTIFY的管程模型也同样可以实现上述功能，但是他们是基于Monitor锁的；Condition是基于可重入锁实现的，可重入锁是基于AQS的。

Condition的signal() signalAll() await()调用对象是Condition实例，不能调用Object的notify() notifyAll() wait()!

调用Object的上述方法必须获取Monitor锁，同理，调用Condition的几个方法也必须获取到相应的锁。

每个ReentrantLock实例可以多次调用newCondition，产生多个ConditionObject实例.

在AQS中可以看到ConditionObject实现了Condition接口,其中还有两个Node对象属性：

很好理解，Condition也有一个阻塞队列。这里按照Doug Lea的说法叫条件队列。

Node中有4个属性：

  1. Node prev - 前驱
  2. Node next - 后继
  3. Thread thread - 线程本体
  4. Node nextWaiter - 用于实现条件队列

条件队列是一个单向链表，每个对象是一个Node，也就是说，Node类中的nextWaiter属性是专门为条件队列准备的。第一个Node叫firstWaiter,最后一个Node叫lastWaiter.

而条件队列中的节点，最后都要转移到阻塞队列，所以它们都是Node实例。

条件队列执行原理大概如下：

  1. 线程调用condition.await()方法，将当前线程包装成node加入到条件队列中，然后阻塞；
  2. 线程调用condition.signal()触发唤醒，会将condition对应的条件队列的队头firstWaiter移到阻塞队列的队尾，等待获取锁。
  3. 当然，这里是最简单的流程。

唤醒后检查中断状态

被唤醒之后，先判断线程是否被中断了，调用checkInterruptWhileWaiting(Node node)方法：

transferAfterCancelledWait(Node node)方法负责判断中断发生的时机。如果是signal之前中断的，返回true,否则返回false.

为什么CAS成功就代表signal之前发生了中断呢？因为在signal中的transferForSignal()方法中，会设置node的waitStatus为0.因此CAS成功代表signal还没有发生，这里要完成加入阻塞队列的动作。如果中断是在signal()之后、入队之前发生的（极其罕见），那么要等待signal入队操作完成之后再继续。

总之，即使发生了中断，节点最后都是要加入阻塞队列的.

那么，退出while循环的方式有两种，一，节点已经被转移到阻塞队列；二，节点被中断了，那么它被唤醒，自己转移到阻塞队列。

附录 java线程的中断和处理

线程的Interrupt是一种状态，可以为true或者false.

Thread中的isInterrupted()实例方法返回线程的中断标记，interrupted()静态方法返回线程是否已经中断，并且将中断标记修改为false，interrupt()实例方法设置一个线程的中断状态为true.

线程处于来自 Object 类的 wait()、wait(long)、wait(long, int)，

来自 Thread 类的 join()、join(long)、join(long, int)、sleep(long)、sleep(long, int)情况的时候被中断，抛出InterruptedException.

实际上，方法声明中有抛出InterruptedException的，基本都是阻塞方法，都会响应该异常。

通常讲，我们可以通过另一个线程中断这些阻塞方法让它们提前返回。在java并发包中通常都有响应中断和不响应中断的方法，比如lockInterruptibly()和lock()，前者被中断后会响应并取消抢锁，而后者仅仅是记录下中断状态。
