JVM基础

发表于 2019-06-12 | 分类于 JAVA
本文字数： 6.3k | 阅读时长 ≈ 6 分钟

本文主要介绍JVM基础知识，包括Java内存区域、类加载过程和类加载器、对象创建、垃圾回收相关知识点。

资料来源：GitHub-JavaGuide、《深入理解JAVA虚拟机-JVM高级特性与最佳实践（第二版）》、个人整理

Java内存区域

Java内存区域分为线程共有和线程私有两部分。

线程共有的是堆和方法区（非堆），线程私有的是程序计数器、本地方法栈、虚拟机栈。

可以简单的讲，堆空间是线程共有的，栈空间是线程私有的。

程序计数器（线程私有）

选取下一条需要执行的字节码指令。

如果执行是Java方法，储存的是当前执行虚拟机字节码指令的地址。如果执行的是Native方法，计数值为空。

虚拟机栈（线程私有）

描述Java方法执行时的内存模型。每个Java方法执行时都会创建一个栈帧，存储局部变量表、操作数栈、动态链接、方法出口信息。，每个方法的调用就对应着栈帧从入栈到出栈的过程。

局部变量表存储编译期可知的基本数据类型、对象引用和returnAddress类型。

如果线程请求的栈深度超过虚拟机允许的深度，抛出StackOverFlowError。如果扩展栈时无法申请到足够的内存，抛出OOM。

本地方法栈

与虚拟机栈类似，不过本地方法栈是为Native方法服务的。HotSpot虚拟机甚至直接把本地方法栈和虚拟机栈合二为一。也会抛出两种错误。

堆（线程共有）

堆空间主要储存对象实例。所有的对象实例和数组都在堆上分配。也是GC的区域。分为新生代和老年代。新生代又分为Survivor、Eden区，这部分到GC详谈。

当堆区域无法再扩展时，会抛出OOM错误。

方法区（线程共有）

又名非堆，为了和堆区分。堆里存储的是已经被虚拟机加载的类信息、常量、静态变量、即时编译器编译过的代码等。GC极少出现，主要是针对常量池的回收和内存卸载。无法扩展区域时抛出OOM错误。

String.intern()用来返回常量池中的某字符串，如果常量池中已经存在该字符串，则直接返回常量池中该对象的引用。否则，在常量池中加入该对象，然后 返回引用。在 HotSpot虚拟机1.7之前的版本，常量池存储在方法区的PermGen Space(永久区)。在jdk1.7之后，字符串常量重新被移到了堆中。

jdk1.7后，常量池也可存储对象的引用地址。

对象的创建

  1. 【检查类的加载】虚拟机遇到new指令时，首先检查常量池中是否能够定位到一个类的符号引用，并且检查这个符号引用代表的类是否已完成加载（解析、初始化），如果没有，必须先执行类的加载。
  2. 【分配内存】类的加载检查通过后，JVM为新对象分配内存。所分配内存的大小在类加载后就可以完全确定。根据堆内存是否规整，可以分为指针碰撞（规整）和空闲列表（不规整）两种方式。GC过程是标记-整理或复制的选用指针碰撞，是标记-清除的选择空闲列表。
  3. 【初始化零值】将分配到的内存空间都初始化为零值。保证对象的实例字段在Java代码中不赋初始值就可以直接使用，程序能访问到对应的零值。
  4. 【设置对象头】例如类的元数据信息、对象的哈希等信息。
  5. 【执行方法】在Java程序中把对象按照程序员的意志初始化。

内存分配并发问题

  1. CAS+失败重试： CAS 是乐观锁的一种实现方式。所谓乐观锁就是，每次不加锁而是假设没有冲突而去完成某项操作，如果因为冲突失败就重试，直到成功为止。虚拟机采用 CAS 配上失败重试的方式保证更新操作的原子性。
  2. TLAB： 为每一个线程预先在 Eden 区分配一块儿内存，JVM 在给线程中的对象分配内存时，首先在 TLAB 分配，当对象大于 TLAB 中的剩余内存或 TLAB 的内存已用尽时，再采用上述的 CAS 进行内存分配。

类加载过程

JVM中的类加载需要经过加载、链接和初始化三个步骤。其中链接可以分为验证、准备和解析三个步骤。

加载是查找字节流，并且据此创建类的过程。加载需要借助classLoader.JVM采用了双亲委派模型， 收到加载请求时，会先将请求转发给父类加载器。

链接是指将创建成的类合并至java虚拟机中，使之能够执行的过程。

初始化是类加载的最后一步。真正执行Java程序代码（字节码），是执行类构造器<cinit>()方法的过程。

类的加载

加载，是指查找字节流，并且据此创建类的过程。主要完成以下三件事情：

  1. 通过全类名获取定义此类的二进制字节流
  2. 将字节流代表的静态存储结构转换为方法区的运行时数据结构
  3. 在内存中生成一个代表该类的Class对象作为方法区这些数据的访问入口

类的连接

创建成的类合并到jvm中，使之能够执行的过程。分为验证、准备以及解析三个阶段。

验证

确保被加载满足jvm的约束条件。

包括文件格式验证、元数据验证、字节码验证、符号引用验证。

准备

为被加载类的静态字段分配内存并设置初始值。（但不进行初始化）这些内存分配在方法区。

  1. 这里进行内存分配的仅限于类的静态变量，不包括实例变量。
  2. 这里设置的初始值“通常情况下”是默认的零值。例如public static int value=111那么value的值在准备阶段初始值就是0.特殊情况比如加上了final关键字，那么准备阶段value的值就是111.

部分java虚拟机还会在此阶段构造其它跟类层次相关的数据结构，比如说用来实现虚方法的动态绑定的方法表。

解析

class文件被加载到jvm之前，这个类无法知道自己方法字段对应的具体地址，所以编译器给他生成一个符号引用。

解析的目的就是将符号引用解析成实际引用。如果符号引用指定一个未被加载的类，或者未被加载类的字段或方法，那么解析将触发类的加载。（但未必触发这个类的链接以及初始化）

初始化就是为标记为常量值的字段赋值，以及执行方法的过程。初始化完成后，类才真正可以执行。

  1. 虚拟机启动时，初始化主类。
  2. 新建目标类实例的New指令，初始化new指令的目标类
  3. 调用静态方法的指令时，初始化该静态方法所在的类
  4. 访问静态字段的指令时，初始化该静态字段所在的类。
  5. 子类初始化触发父类初始化
  6. 反射调用时触发目标类初始化

类加载器

最上层的BootStrap ClassLoader 是用C++实现的，在java中使用null来加载。

在java9之前，BootStrap classLoader负责最基础重要的类，比如jre的lib目录下的jar包中的类。除了bootstrap classLoader之外，另外两个重要的类加载器是扩展类加载器和应用类加载器。扩展类加载器的父加载器是bootstrap classLoader,而应用类加载器的父加载器是扩展类加载器。

当然 我们也可以利用自定义的类加载器实现特殊的加载方式。

双亲委派模型

类加载器需要给父加载器转发加载请求，当父加载器确定没有找到所请求的类的情况下，该类加载器才会尝试加载。

双亲委派模型保证了Java程序的稳定运行，可以避免类的重复加载（JVM 区分不同类的方式不仅仅根据类名，相同的类文件被不同的类加载器加载产生的是两个不同的类），也保证了 Java 的核心 API 不被篡改。如果不用没有使用双亲委派模型，而是每个类加载器加载自己的话就会出现一些问题，比如我们编写一个称为 java.lang.Object 类的话，那么程序运行的时候，系统就会出现多个不同的 Object 类。

判断对象没有被引用（已死）

引用计数

对象有引用计数器，引用对象时计数器加一，引用失效时计数器减一。当计数器为0时代表对象不再被引用，可以被回收。

JVM没有使用这个机制，因为无法解决循环引用问题。

可达性分析

通过一系列GC Roots对象作为起始点，从这些节点向下搜索，搜索走过的路径成为引用链。当一个对象到GC Roots没有任何引用链相连，证明此对象不可用。

Java中可作为GC Roots对象的有以下几种:

  1. 虚拟机栈（栈帧中的本地变量表）中引用的对象
  2. 方法区中类静态属性引用的对象
  3. 方法区中常量引用的对象
  4. 本地方法栈中Native方法引用的对象

强引用 软引用 弱引用 虚引用

强引用

程序代码中显式声明的引用。如果强引用存在，GC永远不会回收掉被强引用的对象。如Object obj = new Object();

软引用

描述一些有用但非必需的对象。对于软引用对象，系统将要发生OOM之前，将会对其列入GC范围之中进行第二次回收。

通过SoftReference类实现软引用。

弱引用

描述强度比软引用更弱一些的非必需对象。GC工作时无论内存是否足够，都会回收掉只被弱引用关联的对象。

通过WeakReference类实现弱引用。

虚引用

最弱的引用关系，完全不会对对象生存时间构成影响。唯一的目的是这个对象被GC时可以收到一个系统通知。

通过PhantomReference类实现虚引用。

回收方法区（HotspotVM中的永久代）

主要针对废弃常量和类。

针对废弃常量，只要没有被引用就可以被回收。

针对类，只有达到以下要求才可能被回收：

  1. 该类在堆中的所有实例都已经被回收
  2. 该类的ClassLoader已经被回收
  3. 该类的Class对象没有被任何地方引用，无法在任何地方通过反射访问该类的方法

GC算法

标记-清除算法

最基础的算法，两个阶段，先标记后清除。

问题1，两个阶段的效率都不高；问题2，清除之后产生碎片，导致内存空间不连续，可能会提前触发下一次GC。

复制算法

每次只使用内存的一半，内存用完后将存活的对象复制到另一块，然后完全清空已经使用过的内存空间。

优点：回收半区内存，不用考虑碎片，按顺序分配内存，简单高效。

缺点：只能利用一半内存。

现在基本都采用这种方式回收新生代，而新生代并非按照1:1划分内存看空间，而是分为较大的Eden空间和较小的两块Survivor空间，每次使用一块Survivor和Eden。回收时将存活对象复制到另外一个Survivor，然后清空用过的Survivor和Eden。

默认Eden和Survivor比例为8:1:1，当某次Survivor空间不足时，通过老年代内存空间担保机制，可以令部分对象直接进入老年代。

标记-整理算法

由于老年代没有额外的内存进行分配担保，因此一般不采用复制算法。标记-整理算法令所有存活对象向一端移动，然后直接清理掉边界以外的内存。这样整理后得到连续的内存空间。

分代收集

根据每个年代的特点采用最适当的算法。

垃圾收集器

收集算法的具体实现。

Serial收集器

单线程， GC时暂停其它所有工作线程。

新生代采用复制算法，老年代采用标记-整理算法。

ParNew收集器

新生代，多线程的Serial收集器版本，可以和CMS配合工作。

Parallel Scavenge收集器

新生代收集器，采用复制算法，多线程并行。

目标是达到一个可控制的吞吐量。可以自适应调节。

Parallel Old收集器

老年代收集器，Parallel Scavenge收集器的老年代版本，采用标记-整理算法，与Parallel Scavenge配合使用。

CMS收集器（Concurrent Mark Sweep）

老年代收集器，以获取最短停顿时间为目标，适合BS系统。

CMS收集器从名字上看出是基于标记-清除算法，它的运作过程主要分为4个步骤：初始标记、并发标记、重新标记、并发清除。

其中，初始标记、重新标记这两个步骤仍然需要STW。初始标记是简单标记一下GC Roots能关联到的对象，速度很快；并发标记阶段就是进行可达性分析的过程；重新标记阶段是为了修正并发标记期间因用户程序继续运作导致标记产生变动的那一部分对象的标记记录，比初始标记阶段稍长，但是远比并发标记短。

整个过程中耗时最长的并发标记和并发清除过程GC线程都可以与用户线程一起工作，总体上看CMS的GC过程是和用户线程并发执行的。

CMS在并发清除时用户还在产生垃圾，因此需要预留空间，在老年代达到92%的时候提前启动。如果运行期间内存无法满足需要，就会出现一次临时启用Serial Old收集器重新进行老年代的垃圾收集。

CMS采用的标记清除算法会出现没有足够大的连续空间分配对象问题，为此提供了进入FULLGC后开启碎片整理过程的开关。

G1收集器（Garbage First）

G1收集器，把堆空间分成一个个大小相等的Region，整体上看基于标记-整理算法，局部看基于复制算法。这意味着G1运作时间不会产生内存中心碎片，不会提前触发GC。

G1的“新生代”和“老年代”没有物理隔离，都是一部分Region的集合。G1可以实现“可预测的停顿”，即指定M毫秒内运行GC的时间为N毫秒。G1通过跟踪Region里面的垃圾价值大小，在后台维护优先列表，每次根据允许的收集时间，优先回收价值最大的Region。通过Remember Set来避免全堆扫描。

G1收集器的运作大致分为以下几个步骤：初始标记、并发标记、最终标记、筛选回收。

其中除并发标记外，都需要STW（筛选回收可以并发，但是效率没有STW后并行的效率高）。

  1. 初始标记阶段标记一下GC Roots能直接关联到的对象，并且修改TAMS的值，让下一阶段用户程序并发运行时能在正确可用的Region中创建新对象。
  2. 并发标记阶段，做可达性分析，用户程序修改的标记记录，记录到Remember Set Logs里面
  3. 最终标记阶段，把Remembered Set Logs里的数据合并到Remembered Set中，可并行执行
  4. 筛选回收阶段，对各个Region的回收价值和成本排序，根据用户期望的GC停顿时间来制定回收计划。

G1 参数配置和最佳实践

G1 调优的目标是尽量避免出现 Full GC，其实就是给老年代足够的空间，或相对更多的空间。

有以下几点我们可以进行调整的方向：

  • 增加堆大小，或调整老年代和年轻代的比例，这个很好理解
  • 增加并发周期的线程数量，其实就是为了加快并发周期快点结束
  • 让并发周期尽早开始，这个是通过设置堆使用占比来调整的（默认 45%）
  • 在混合垃圾回收周期中回收更多的老年代区块
